🔴 Issue 1: contentPillars extraction is underdefined
You wrote: "extracted from rawSections → brand_voice". This is hand-wavy. rawSections is (I assume) unstructured text blob. How does extraction work?

Three options, choose one:

Dedicated DB field — add a contentPillars column to brands that BrandSmith fills during brand setup. Clean but requires BrandSmith change.
LLM pre-processing — before calling the miner, Replit makes its own Claude call: "Given this brand voice section, list 4-6 content pillars as a JSON array." Adds ~$0.005 and ~2s per request. Clean but one extra moving part.
Regex/heuristic split — split the brand_voice text on newlines/bullets, first 6 items become pillars. Zero cost, very brittle.
My recommendation: Option 2 (LLM pre-processing) for v1 if BrandSmith can't add a field, then migrate to Option 1 when BrandSmith supports it. Reason: Option 3 will produce garbage pillars which will produce garbage drafts, and pillar quality directly affects draft quality.

Without good pillars, the drafts fall back to generic brand-voice inference. With good pillars, drafts are on-strategy.

🟡 Issue 2: voiceRules and noGoTopics being empty
You have voiceRules: [] and noGoTopics: []. Acceptable but will degrade draft quality noticeably.

With empty voice rules the LLM has no hard constraints. It'll produce "safe" drafts in a generic marketing voice. With rules like "Never say 'seamless'" or "Use first-person plural" the LLM sticks to brand voice much better.

Three options:

Empty for v1, add later — acceptable. Drafts will be OK but generic.
Manual advanced fields in Replit UI — collapsible "Advanced voice rules" section where power users can add rules. Most users ignore it, power users use it.
Auto-generate from voiceArchetype — one-shot Claude call converting "playful-chaotic-honest" into 3-5 voice rules. Same idea as Option 1 above for pillars.
My recommendation: Empty for v1, add optional manual fields in v1.1 after you see first-run quality. Don't block v1 on this.

🟡 Issue 3: brand.id = website_url is fragile
Using URL as primary identifier has problems:

URLs change (rebrands, domain moves)
Contains protocol (https://), trailing slashes, www. prefix inconsistencies
Harder to use in URLs, logs, CLI commands
Better: Use a stable slug derived from brand name (tryps, brand-smith, dominos-india) or a UUID. The miner doesn't care what the ID is — it's just echoed back in runMetadata.theme. But for logging/debugging/support it should be human-readable and stable.

My recommendation: Use slug(brand.name) — e.g., "TRYPS" → "tryps". Or a UUID if you want perfect stability across renames. Either way, avoid website_url as the key.

🟡 Issue 4: Your Quick mode skips too much
Your plan: Quick = nano: true, everything else false.

Problem: trending modes cost ~5 quota each and take ~5 seconds each. They're basically free. Skipping them saves nothing but loses a ton of signal (in-niche trending context + cross-niche format inspiration).

Better Quick mode definition:

Quick scan (~40s, ~$0.08):
  nano:              true
  mid:               false   ← the expensive one to skip (500 quota + 20s)
  trendingInNiche:   true    ← cheap (~5 quota, ~5s)
  trendingCrossNiche: true   ← cheap (~5 quota, ~5s)
Full scan (~90s, ~$0.15):
  nano:              true
  mid:               true    ← adds ~500 quota + 20s for mid-tier validation
  trendingInNiche:   true
  trendingCrossNiche: true
Why: Trending modes are the format-inspiration layer. Users doing "quick iterations" still benefit hugely from cross-niche format signal. You only skip mid tier because it's the one expensive mode that adds validation value but not ideation value.

This change makes Quick mode ~35-40s instead of ~30s (barely slower) but gives the user ~10x more signal for format borrowing. Trading 10 seconds for 10x signal is a no-brainer.

🟡 Issue 5: Missing requestId in your payload
You should include a client-generated UUID as requestId in the POST body. Reasons:

Idempotency — if request times out and Replit retries, same ID means miner can return cached result (if we add caching)
Logging — correlate Replit UI state with miner server logs when debugging
Observability — trace requests through the whole pipeline
Trivial to add. One UUID per submit click.

🟢 Issue 6: "Stored locally" is ambiguous
You wrote "Full JSON back → stored locally → rendered". Where?

React state (lost on refresh)
localStorage (survives refresh, lost when browser data cleared)
Replit DB (persistent, enables history view later)
For v1: React state is fine. User runs a plan, sees results, uses them. If they refresh they lose it — acceptable for v1.

For v1.1: persist to Replit DB so you get a "past runs" history tab. Each brand has a log of plans over time. Useful for tracking what's been tried.

My recommendation: React state for v1, plan for DB persistence in v1.1.

🟢 Issue 7: No progress indicator beyond elapsed timer
60-90 second wait with only a spinner + timer is mediocre UX. Users will wonder if it's frozen.

Better options (in order of effort):

Stage messages — rotate through "Searching YouTube... Extracting transcripts... Analyzing patterns... Generating drafts..." every 15 seconds. Fake progress but reassuring.
Estimated countdown — "Estimated 60 seconds remaining". Requires a server-provided estimate.
Server-Sent Events — miner pushes progress updates to Replit, Replit forwards to browser. Real progress. More work to build (~2 hours extra).
My recommendation: Start with Option 1 (stage messages) for v1. SSE in v1.2 if users complain.

🟢 Issue 8: Error state handling not spec'd
What does the UI show when:

401 (shouldn't happen in prod, but does in dev)
429 (quota exhausted)
500 (server error)
Network timeout (>120s)
Replit should handle each with a specific message:

401 → "Authentication failed — contact admin"
429 → "Daily YouTube quota exhausted. Try again tomorrow or reduce scope."
500 → "Something broke on our end. Error ID: {requestId}. Retry in a moment."
Timeout → "Request took too long. Try with fewer seeds or Quick mode."
These go in your error-handling layer, not the happy path, but worth specifying now.
