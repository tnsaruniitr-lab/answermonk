🔴 Where the spec diverges from our conversation (matters)
1. The "wow factor" preview layer isn't in the spec
We discussed at length: The preview output pattern — 5 critical gaps + rankings grid (Google × ChatGPT × Perplexity × Claude) + authority signals comparison table + competitor anxiety line + "what you're good at" line + category benchmarks. The entire conversion-optimized preview structure from the launch brief.

Spec coverage: Missing entirely. §12 UI describes two tabs (Authority + Social matrices) and that's it. Nothing about:

A headline "5 critical gaps" extraction layer
A rankings grid showing citation gaps across 4 AI engines
A "you vs. category benchmark" presentation
Outcome prediction framing (even as an v2 intent)
Visibility band framing (Strong/Moderate/Weak)
Why this might be intentional: The spec is titled "AI Visibility Authority Extension" — as a backend extension to an existing scoring pipeline. The preview + gap presentation may belong in a separate spec about the report UX. §18 does explicitly exclude "Shareable preview images" and "Email capture gating" from v1.

Risk: If the spec is the only planning document, the build will produce a matrix UI with no compression into the customer-facing findings layer. The matrix is the data source; the preview is the product. Shipping just the matrix ships 70% of the work and 20% of the customer value.

Recommendation: Either (a) add a §12.5 "Preview extraction layer" — take the matrix, identify top 5 gaps, render as findings cards — or (b) confirm there's a companion spec for that.

2. "5 critical gaps" extraction logic isn't specified
Even within the Authority + Social tabs, the spec describes the matrix but not how to compress it into findings. The conversation had a "5 specific, named findings with problem + gap + fix + effort + priority" pattern.

Spec gap: No logic for "given a 6×5 authority matrix + 6×6 social matrix, which 5 cells are the biggest gaps?" This is the key compression step between raw data and customer-readable insight.

Why it matters: Without this logic, customers see a 36-cell grid and have to parse it themselves. That's the opposite of what we agreed on ("users look at 10 seconds max per finding").

3. Rankings grid (spec vs. existing feature)
We discussed: A 3-query × 4-engine grid (Google, ChatGPT, Perplexity, Claude) showing brand-citation status per query — the centerpiece of the preview that creates the visceral "AI can't find you" moment.

Spec treatment: §2 says the existing system already scores across ChatGPT, Gemini, Claude. Missing: Google SERP position as the comparison baseline. Also missing: Perplexity (spec only lists ChatGPT/Gemini/Claude as the scoring engines).

Why this matters: The conversion pitch for the preview ("you rank #1 on Google but invisible on ChatGPT") requires all four — Google ranking AND the three AI engines side-by-side. The existing AnswerMonk tracks 3 AI engines but apparently not Google SERP position. Unless that already exists elsewhere, it's a gap.

4. "Bands not scores" framing didn't land
We discussed: Present confidence as Strong/Moderate/Weak bands at launch, move to precise scores only when methodology is public. The "43%" precision implies objectivity evidence doesn't support.

Spec: §18 excludes "Numeric visibility scores with precise decimals" — ✅ correctly excluded. But §12 Authority tab UI says "You appear on X of Y authority signals vs. category average Z" — which is a precise number, not a band. And the tier badges (HIGH/MID/LOW per cell) are per-cell, not overall.

Not a contradiction, but worth naming. The spec is consistent internally; it just doesn't adopt the Strong/Moderate/Weak overall-band framing. Probably fine — per-cell tiers are arguably more useful than a one-band summary. But decide consciously.

5. Category medians — the seed-vs-observed problem
We discussed: Don't show category medians until you have 30+ audits per category; use ranges ("between 200 and 500 Reddit mentions") until then.

Spec §10: category_medians table has median_value, p25_value, p75_value. Implies numeric precision from day 1.

Spec §18 partial save: "Category medians — seed manually, compute once 30+ audits done." So the intent is right. But the table schema stores a median_value field which will be misleading if populated with manual estimates on day 1. Fix: add an is_manual_estimate: boolean column, or use low_bound/high_bound instead of p25/median/p75.
