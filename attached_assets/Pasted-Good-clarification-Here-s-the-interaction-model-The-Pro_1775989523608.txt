Good clarification. Here's the interaction model:

The Problem with a Single HTTPS Call
The research pipeline (crawl + Perplexity + Claude) takes 30-60 seconds. A single synchronous request has issues:

HTTP timeouts (Replit, proxies, browsers typically cut at 30s)
No progress feedback — user stares at a spinner
If connection drops, all work is lost
Two Options
Option A: Async (2-call pattern) — recommended

1. POST /api/brands/research  →  { job_id, status: "pending" }     (instant)
2. GET  /api/brands/research/:jobId  →  { status: "researching", progress: "Analyzing competitors..." }  (poll every 3s)
3. GET  /api/brands/research/:jobId  →  { status: "ready" }        (done!)
4. GET  /api/brands/:jobId/cards  →  7 card sections               (instant)
5. PUT  /api/brands/:jobId/cards  →  save edits                    (instant)
6. POST /api/brands/:jobId/confirm  →  save to Supabase            (instant)

Option B: SSE stream (single call, real-time)

1. POST /api/brands/research  →  SSE stream that emits progress events, then final cards payload
   event: progress  data: { step: "crawling", message: "Scanning pricing page..." }
   event: progress  data: { step: "researching", message: "Analyzing competitors..." }
   event: complete  data: { cards: [...7 sections...] }
2. PUT  /api/brands/:jobId/cards  →  save edits
3. POST /api/brands/:jobId/confirm  →  save to Supabase

SSE is still a single HTTPS connection — the server just keeps it open and pushes events. No WebSocket complexity. The Replit frontend uses EventSource or fetch with streaming.
